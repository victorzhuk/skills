# Clean output

Write every artifact as a human maintainer would, with no trace of the process that produced it.

## Core rules

- Never mention AI tools, models, or assistants in code, docs, comments, configs, commits, or changelog entries.
- Never state that code was "generated" or "AI-assisted," and never leave meta-commentary about the generation process.
- Never cite an internal rule name, prompt, or instruction you're following inside the artifact itself — the artifact should stand on its own.
- Exception: planning artifacts that need traceability (change proposals, task briefs, test plans, QA reports, internal review reports) may cite real tracker/ticket IDs — that's operator-facing traceability, not process metadata.
- Diagnostic reports may include evidence and verification commands; put tool/process detail in a debug appendix only when the artifact's audience explicitly wants that.

## Examples

**Avoid:**

```go
// Generated by AI
// TODO: AI to refine this later
// Following the no-comments rule
```

```
Per the spec, implemented the retry logic.
```

**Write instead:**

```go
// Legacy clients depend on this header until the v2 rollout completes
```

```
Add retry with exponential backoff for gateway timeouts.
```

## Do not

- Leave a code comment or commit message that references a prompt, spec ID, or rule file by name.
- Note "this was auto-generated" on code you wrote as part of the task, unless the file is genuinely tool-generated (codegen output, where a generated-file marker is the correct and expected convention).
- Explain your reasoning process inside the artifact instead of in your response to the user.

## Verify

    rg -ni 'generated by|ai-assisted|per the (rule|spec)|following the .* rule' -- <changed files>
