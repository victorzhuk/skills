# Go database access

No ORMs. SQL is explicit, visible, and testable. House default is `pgx`/`pgxpool`
with queries generated by `sqlc` — see [[z-go-sqlc]] for the codegen side.

## Library choice

| | `database/sql` | `sqlx` | `pgx` |
|---|---|---|---|
| Multi-database | yes | yes | no |
| Struct scanning | manual `Scan` | `StructScan` / `Select` | `pgx.RowToStructByName`, or generated by `sqlc` |
| PG-specific (COPY, LISTEN, arrays) | no | no | yes |
| Performance | baseline | ~baseline | 30–50% faster |

Pick `pgx` for PostgreSQL-only services — the house default, paired with
`sqlc`-generated queries. Pick `sqlx` only when multi-database portability is
a real requirement.
Never GORM/ent — unpredictable query generation, N+1 by default, leaky abstractions.

## Parameterized queries

```go
// bad — SQL injection
query := fmt.Sprintf("SELECT * FROM users WHERE email = '%s'", email)

// good — pgx, parameterized
var u User
err := pool.QueryRow(ctx, "SELECT id, name FROM users WHERE email = $1", email).
    Scan(&u.ID, &u.Name)
```

In practice these are `sqlc`-generated calls (`q.GetUserByEmail(ctx, email)`),
not hand-written — see [[z-go-sqlc]]. The raw form above is what runs
underneath, and still applies to ad-hoc queries outside sqlc's reach.

Dynamic `IN`: pgx encodes a Go slice as a native Postgres array, so pass it
straight through — no query-rewriting helper needed:

```go
rows, err := pool.Query(ctx, "SELECT * FROM users WHERE id = ANY($1)", ids)
```

(`sqlx` lacks native array support and needs `sqlx.In` + `db.Rebind` to expand
a slice into placeholders — only relevant if you're on the `sqlx` path.)

Dynamic sort column: validate against an allowlist (`map[string]bool`), then `fmt.Sprintf` — never pass raw user input.

## NULLable columns

`sqlc` generates the scan-safe type for a NULLable column automatically
(pointer or `pgtype.X`, depending on `sqlc.yaml`) — see [[z-go-sqlc]]. Mapping
by hand, use pointer fields — they scan correctly and marshal to JSON `null`:

```go
type User struct {
    ID        int64
    Name      string
    DeletedAt *time.Time // NULL-safe
    Bio       *string
}
```

Avoid `sql.NullString` / `sql.NullTime` — `Valid bool` leaks persistence into domain types.

## Error handling

```go
var u User
err := pool.QueryRow(ctx, "SELECT id, name FROM users WHERE id = $1", id).
    Scan(&u.ID, &u.Name)
if errors.Is(err, pgx.ErrNoRows) {
    return nil, ErrUserNotFound // translate to domain sentinel
}
if err != nil {
    return nil, fmt.Errorf("get user %s: %w", id, err)
}
```

Always `defer rows.Close()` and check `rows.Err()` after the loop. Use `pool.Exec` for writes — `pool.Query` opens `Rows` that leaks the connection if not closed.

Key PG error codes: `23505` unique violation → conflict sentinel; `40001`
serialization failure → retry; `context.Canceled` → propagate up.

## Transactions

```go
tx, err := pool.BeginTx(ctx, pgx.TxOptions{})
if err != nil {
    return fmt.Errorf("begin tx: %w", err)
}
defer func() {
    if p := recover(); p != nil {
        _ = tx.Rollback(ctx)
        panic(p)
    } else if err != nil {
        _ = tx.Rollback(ctx)
    }
}()

// ... operations on tx ...

if err = tx.Commit(ctx); err != nil {
    return fmt.Errorf("commit: %w", err)
}
```

### Isolation levels

```go
tx, err := pool.BeginTx(ctx, pgx.TxOptions{
    IsoLevel: pgx.Serializable,
})
```

Use `Serializable` for financial operations; default `ReadCommitted` for most reads.
On `40001` (serialization failure) retry the entire transaction from scratch.

### SELECT FOR UPDATE

```go
var o Order
err := tx.QueryRow(ctx, "SELECT id, status FROM orders WHERE id = $1 FOR UPDATE", id).
    Scan(&o.ID, &o.Status)
```

Locks the row until commit — prevents concurrent writes between read and update. Non-blocking variant: `FOR UPDATE NOWAIT`.

## Connection pool

```go
cfg, err := pgxpool.ParseConfig(dsn)
if err != nil {
    return nil, fmt.Errorf("parse pool config: %w", err)
}
cfg.MaxConns = 25
cfg.MinConns = 5
cfg.MaxConnLifetime = time.Hour
cfg.MaxConnIdleTime = 30 * time.Minute

pool, err := pgxpool.NewWithConfig(ctx, cfg)
```

`MaxConns` ≤ PG `max_connections` / instances. Left unset it defaults to
`max(4, NumCPU)` — size it explicitly for production load rather than relying
on the default.

(`database/sql`/`sqlx` pools use `db.SetMaxOpenConns`/`SetMaxIdleConns`/
`SetConnMaxLifetime`/`SetConnMaxIdleTime` instead — same knobs, different API,
only relevant off the pgx path.)

## Migrations

External tool only; never hand-roll or generate SQL with AI. Default to `goose`
(embedded SQL + Go migrations, runs from an `embed.FS` or CLI) — see [[z-go-goose]].
Alternatives: `golang-migrate` (CLI + library) for non-Postgres databases, `Atlas`
(declarative + drift detection) for schema-as-state. Files go in source control.

Prefer running migrations from CI/CD over application startup. A
startup-gated migration behind an explicit opt-in env flag (e.g.
`RUN_MIGRATIONS_ON_STARTUP`, defaulting to a safe choice) is an accepted
pattern here, not a violation — gate it, don't run it unconditionally.

## Do not

- Interpolate user input into SQL — always parameterize.
- Use `pool.Query` for writes; use `pool.Exec`.
- Omit `rows.Close()` or `rows.Err()` — silent leaks and partial results.
- Drop the context on a query call — pgx takes it as the first parameter on
  every method; on `sqlx`, always use the `...Context` variant.
- Rely on triggers, views, or stored procedures — invisible side effects.
- Generate or modify schemas from application code — migrations only.
- Run migrations unconditionally from `main()` — gate behind an explicit
  opt-in flag, or run from CI/CD.
